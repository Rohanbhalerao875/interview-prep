# DSA Review Policy

## Core rule
- Preserve the user's written solution logic and structure unless the user explicitly asks to change the algorithm.
- Do not replace the user's code with a different approach just because a more optimized version exists.
- If a better approach exists, give it as a hint in a separate place such as the follow-up tracker or a short note, rather than rewriting the main solution.

## What is allowed
- Improve formatting, indentation, naming consistency, and readability.
- Add brief comments only when they help the user review the code later.
- Suggest better approaches as hints, follow-up ideas, or optional notes.

## What is not allowed without explicit request
- Rewriting the core logic of the user's solution.
- Replacing their approach with a different algorithm in the main file.
- Changing the code flow unless the user asks for it.

## Review style for DSA problems
- Keep the user's original thinking visible.
- If a better approach exists, say: "Hint: this can also be solved using ..." and place that guidance in the follow-up tracker or in a small note section.
- Prefer small, non-invasive improvements so the user can still recognize their own work.

## Repo workflow
- When a solution is updated, keep the solution file and the trackers consistent.
- Use the follow-up tracker for advanced variations and future practice.
